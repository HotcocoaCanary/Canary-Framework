# 架构哲学与核心引擎

Canary Framework 遵循高度解耦的**微内核与插件化**（Micro-kernel & Plugins）三层架构：

```
 common/  ──►  decorators/  ──►     core/      ──► canary.py (容器启动器)
(类型,错误)      (元数据标注)        (DI引擎注册表)   
                                      └──► web/ (HTTP内建扩展)
```

## 设计概述

1. **核心逻辑（Core Engine）**：不包含任何业务逻辑，只负责生命周期管理、类的元数据解析，以及通过 Kahn 算法实现无缝的依赖注入。
2. **纯粹的容器**：整个框架以 `Canary` 类作为入口。你只需要提供一个被 `@module()` 标注的入口根模块，容器会自动接管它及其所有子服务。
3. **插件系统（Plugins/Web）**：HTTP 路由、API 文档生成等均不再是核心引擎强制绑定的特性，它们作为官方的内建扩展被内聚在 `core/web/` 下。如果你只想用 Canary 写一个 CLI 工具或无头 Agent 后台，你完全不与 Web 发生耦合。

## 容器启动流程 (Container Bootstrapping)

当你执行 `Canary(AppModule())` 时，引擎会发生以下动作：
1. **依赖收集**：自顶向下扫描 `AppModule` 以及所有被声明的 `services`。
2. **构建依赖图**：扫描所有类上的类型注解（Type Hints），将引用关联。
3. **拓扑排序**：运行 Kahn 算法，验证是否存在循环依赖，并排布出一条无错的实例化顺序。
4. **服务实例化与自动注入**：按照拓扑顺序依次 `cls()`，并通过 `setattr` 将依赖的实例注入到所需的对象中。
5. **初始化挂载**：触发每个服务上的 `init()` 方法（如果存在）。
6. **Web 适配 (可选)**：如果检测到服务具有 `Router` 实例，则开始将它们统一打平成原生的 ASGI 路由列表。

## 为什么是纯装饰器？

旧版框架（或许多传统框架）要求开发者必须继承诸如 `ServiceBase` 或 `ModuleBase` 这样的基类。这严重违反了控制反转原则（IoC），导致用户代码被框架“绑架”。
现在，你的代码完全是原生的 Python 类，没有任何框架继承包袱。框架对你的代码做到真正的**零侵入**。
